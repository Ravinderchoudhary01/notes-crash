UNIT 1

Introduction to Compilers

Definition • Phases • Compiler vs Interpreter • Language Processors • Structure

1.1 Introduction

A compiler is a special program that translates a program written in one language (the source language) into an equivalent program in another language (the target language), usually machine code or assembly. The translation process is called compilation, and the software tool performing it is the compiler.

📌 Compiler:

A program that reads a program written in one language — the source language — and translates it into an equivalent program in another language — the target language. (Aho, Lam, Sethi, Ullman — Dragon Book)

📌 Source Language:

The high-level programming language in which the original program is written (e.g., C, C++, Java, Python).

📌 Target Language:

The low-level language (usually machine code or assembly) into which the source program is translated.

1.2 Language Processors

Programs written in high-level languages must be translated before execution. Different tools accomplish this translation in different ways:

ToolDescriptionExample
CompilerTranslates entire source code to target code before execution; fast executionGCC (C), javac (Java), g++ (C++)
InterpreterTranslates and executes source code line-by-line; no separate object filePython, Ruby, BASIC interpreters
AssemblerTranslates assembly language to machine codeNASM, MASM, GAS
LinkerCombines multiple object files into one executableGNU ld, Microsoft LINK
LoaderLoads executable from disk into main memory for executionOS-level program loader
PreprocessorPerforms macro substitution, file inclusion before compilationC Preprocessor (cpp)
Cross CompilerCompiles code for a different target platform than the hostARM cross-compiler on x86 PC

1.3 Compiler vs Interpreter

FeatureCompilerInterpreter
TranslationEntire program at onceLine by line
Execution SpeedFaster — machine code runs directlySlower — translates at runtime
Error DetectionAll errors after full analysisStops at first error encountered
OutputSeparate object/machine code fileNo separate output file
MemoryMore memory for object codeLess memory — no object file stored
PlatformPlatform-specific object codePlatform-independent if interpreter present
ExamplesC, C++, FORTRAN, COBOLPython, Ruby, JavaScript (traditional)
DebuggingHarder (indirect)Easier (interactive)
🧠 MEMORY TRICK:

'CIA' — Compiler = Instant execution after All translation. Interpreter = Immediate line translation.

1.4 Phases of a Compiler

A compiler operates in a series of phases. Each phase transforms the source program from one representation to another. The phases are broadly grouped into the Front End (machine-independent, source analysis) and Back End (machine-dependent, target synthesis).

5. Code OptimizationMiddleIROptimized IR
6. Code GenerationBack EndOptimized IRTarget machine code
7. Symbol Table ManagerSupportAll phasesSymbol table (cross-phase)
8. Error HandlerSupportAll phasesError messages

1.5 Symbol Table

The Symbol Table is a data structure maintained throughout compilation that stores information about identifiers (variable names, function names, class names, etc.).

Attribute StoredDescriptionExample
Name (Lexeme)The actual identifier stringposition, rate, initial
TypeData type of the identifierint, float, char, pointer
ScopeWhere the identifier is visibleglobal, local, function-level
Memory LocationAddress or offset in memoryoffset 0x0004 in stack frame

💡 Key Insight:

The Symbol Table is NOT a compiler phase — it is a shared data structure used by ALL phases. It is queried during lexical analysis, updated during semantic analysis, and used during code generation.

UNIT 2

Lexical Analysis

Tokens • Regular Expressions • DFA/NFA • LEX Tool • Transition Diagrams

2.1 Introduction

Lexical Analysis (Scanning) is the first phase of compilation. The lexical analyser reads the source program character by character from left to right and groups them into meaningful sequences called lexemes. For each lexeme, the lexical analyser produces a token as output.

📌 Lexical Analyser

The part of the compiler that breaks the source program into a stream of tokens. It removes whitespace, comments, and irrelevant characters.

📌 Token

A pair of the form <token-name, attribute-value>. The token-name is an abstract symbol representing a lexical category (e.g., id, number, operator).

📌 Lexeme

The actual character sequence that matches the pattern for a token. e.g., 'position' is a lexeme for token <id, 1>.

📌 Pattern

A description (often using regular expressions) of the form that lexemes of a token may take.

2.2 Token, Lexeme & Pattern — Examples

Token NameSample LexemesPattern (Informal)
ififThe keyword 'if'
elseelseThe keyword 'else'
comparison<, >, ==, <=, >=, !=Any relational operator
idposition, rate, initialLetter followed by letters/digits
number60, 3.14, 0.01Integer or floating-point constant
literal"hello world"Anything except " between two "
assignment=The = sign
ws(blank) (tab) (newline)Any whitespace character sequence

2.3 Regular Expressions (RE)

Regular expressions are a formal notation for specifying patterns (sets of strings). The lexical analyser uses regular expressions to specify the pattern for each token.

RE OperatorNotationMeaningExample
Concatenationxyx followed by yab means 'ab' only
Union (Alternation)x | yx or ya|b means 'a' or 'b'
Kleene Starx*Zero or more xa* means ε, a, aa, aaa, ...
One or Morex+One or more xa+ = aa* means a, aa, aaa, ...
Optionalx?Zero or one xa? means ε or a
Character Class[xyz]Any of x, y, or z[a-z] = any lowercase letter
Complement[^xyz]Any character except x, y, z[^0-9] = non-digit
Escape\n, \tNewline, tab literal\n is newline character

💡 RE for Identifiers: letter → [A-Za-z_]   digit → [0-9]   id → letter (letter | digit)*

💡 RE for Integers: digit → [0-9]   integer → digit+

💡 RE for Floats: float → digit+ . digit* | . digit+

2.4 Finite Automata

Finite Automata are mathematical models used to recognise regular languages. They are the theoretical basis for implementing lexical analysers.

2.4.1 Deterministic Finite Automaton (DFA)

In a DFA, for each state and each input symbol, there is exactly one transition to the next state. No ambiguity in transitions.

ComponentDescription
Set of States (Q)Finite set of states the automaton can be in
Input Alphabet (Σ)Finite set of input symbols (e.g., {0,1} or ASCII characters)
Transition Function (δ)δ: Q × Σ → Q — maps (current state, input symbol) to next state
Start State (q₀)The state in which the automaton begins processing
Accepting States (F)A subset of Q; if machine ends in one of these, the string is accepted

2.4.2 Nondeterministic Finite Automaton (NFA)

In an NFA, for a state and input symbol, there can be zero, one, or more transitions. NFAs can also have ε-transitions (transitions without consuming input).

FeatureDFANFA
Transitions per (state, input)Exactly oneZero, one, or many
ε-transitionsNot allowedAllowed
Recognition powerSame as NFA (both recognise RLs)Same as DFA
Implementation complexityDirect and efficientRequires subset construction
Number of states (typically)Potentially more (subset construction)Fewer, more compact
Use in compilersFinal implementation (efficient)Intermediate step in construction
🧠 MEMORY TRICK:

'DFA = Direct/Determined' (one path), 'NFA = Non-determined/Multiple paths'. Both recognise EXACTLY the same set of languages (Regular Languages)!

2.5.1 Thompson's Construction (RE → NFA)

RE ConstructNFA Construction Rule
Single symbol 'a'Start state → (on 'a') → Accept state; single transition
ε (empty string)Start state → (on ε) → Accept state
r | s (Union)New start state with ε-transitions to start of both r and s NFAs; merge accepts
rs (Concatenation)Accept of r connected by ε to start of s; r's accept becomes non-accept
r* (Kleene Star)New start/accept state; ε-transitions: start→r, end-of-r→start, start→accept

2.5.2 Subset Construction (NFA → DFA)

StepAction
1. ε-closureCompute all NFA states reachable from a state using only ε-transitions
2. Start StateDFA start state = ε-closure({NFA start state})
3. TransitionsFor each DFA state S and input a: new DFA state = ε-closure(move(S, a))
4. Accept StatesDFA state is accepting if it contains at least one NFA accepting state
5. RepeatContinue until no new DFA states are discovered

UNIT 3

Syntax Analysis (Parsing)

CFG • Parse Trees • Top-Down • Bottom-Up • LL(1) • LR • YACC

3.1 Introduction

Syntax Analysis (Parsing) is the second phase of the compiler. The parser takes the token stream from the lexical analyser and groups tokens into grammatical phrases as defined by the grammar of the programming language. It produces a parse tree or an abstract syntax tree (AST).

📌 Parser:

A program that takes a sequence of tokens and determines whether it can be generated by a grammar (syntactic analysis). If yes, it builds a parse tree; otherwise it reports syntax errors.

3.2 Context-Free Grammar (CFG)

📌 CFG Definition:

A CFG G = (V, T, P, S) where V = non-terminals, T = terminals (tokens), P = production rules, S = start symbol.

CFG ComponentDescriptionExample
Non-terminals (V)Variables that represent syntactic categoriesE, T, F, stmt, expr
Terminals (T)Actual tokens/symbols in the languageid, +, *, (, ), num
Production Rules (P)Rules of the form: A → α (non-terminal → string)E → E + T | T
Start Symbol (S)The non-terminal from which derivation beginsE (for expressions)

3.2.1 Derivation, Parse Trees, and Ambiguity

TermDefinition
DerivationA sequence of replacements of non-terminals using production rules to produce a terminal string
Leftmost DerivationAlways replace the leftmost non-terminal; used by top-down parsers
Rightmost DerivationAlways replace the rightmost non-terminal; used by bottom-up parsers (in reverse)
Parse TreeA graphical representation of a derivation; leaves = terminals, interior nodes = non-terminals
Ambiguous GrammarA grammar where a string has more than one parse tree
Dangling Else ProblemClassic ambiguity in if-then-else constructs; resolved by associating else with nearest if
💡 Classic Expression Grammar:

E → E + T | T   T → T * F | F   F → (E) | id
This grammar correctly handles precedence (* over +) and associativity (left).

3.3 Elimination of Ambiguity, Left Recursion & Left Factoring

ProblemWhy It MattersSolution
Ambiguous GrammarParser can't decide which parse tree to build; non-deterministicRewrite grammar; use precedence/associativity rules
Left Recursion (A → Aα)Causes top-down parsers to loop infinitelyEliminate: A → βA' A' → αA' | ε
Left FactoringMultiple productions with same prefix cause parser confusionFactor: A → αβ | αγ becomes A → αA' A' → β | γ

3.4 Top-Down Parsing

ApproachDescriptionComplexity
Recursive DescentEach non-terminal → one procedure; procedure calls mirror grammar rulesO(n) for LL(1)
Predictive Parsing (LL(1))Uses a parsing table to select production without backtrackingO(n)
Backtracking ParserTries all alternatives; backtracks on failureExponential — impractical

3.4.1 LL(1) Parsing

LL(1) means: Left-to-right scan, Leftmost derivation, 1 token of lookahead. LL(1) parsers use a parsing table constructed from FIRST and FOLLOW sets.

ConceptDefinitionComputation Rule
FIRST(α)Set of terminals that begin strings derivable from αFIRST of terminal = {terminal}; FIRST of non-terminal = union of FIRST of its production bodies
FOLLOW(A)Set of terminals that can appear immediately after A in a sentential formFOLLOW of start symbol includes $; for A→αBβ: FIRST(β)-{ε} ⊆ FOLLOW(B); if β⇒ε, FOLLOW(A) ⊆ FOLLOW(B)
LL(1) Table M[A, a]= production to use when non-terminal is A and lookahead is aFor A→α: add A→α to M[A,a] for each a in FIRST(α); if ε in FIRST(α), add for each b in FOLLOW(A)
🧠 MEMORY TRICK:

'LL First-Follow Foundation' — LL parsers need FIRST sets for direct selections and FOLLOW sets for ε-productions!

3.5 Bottom-Up Parsing

TermDefinition
HandleA substring of the current sentential form that matches the right side of a production and can be reduced
ShiftPush the current input token onto the parser stack
ReducePop the handle off the stack, replace with the corresponding non-terminal
Shift-Reduce ConflictParser can't decide whether to shift or reduce at a given step
Reduce-Reduce ConflictTwo different productions could both apply for a reduction

3.5.1 LR Parsing — Family Overview

LR TypeFull NamePowerComplexity
LR(0)LR with 0 lookaheadWeakest LR; handles limited grammarsSimplest table
SLR(1)Simple LR with 1 lookaheadUses FOLLOW sets to resolve conflictsModerate
LALR(1)Lookahead LR(1)Merges LR(1) states with same core; yacc/bison uses thisPractical balance
LR(1)Canonical LR with 1 lookaheadMost powerful; handles all LR(1) grammarsLarge table — rarely used directly
💡 Parser Power:

LR(0) ⊂ SLR(1) ⊂ LALR(1) ⊂ LR(1). Every LL(1) grammar is also LALR(1). LALR(1) is the standard for compiler tools.

3.5.2 LR Parsing Algorithm (Conceptual)

TableContentUse
ACTION[s, a]shift t / reduce A→β / accept / errorDetermines next action for state s with terminal a on input
GOTO[s, A]New state t after reducing to non-terminal AAfter reducing with A, which state to go to from state s

3.6 YACC (Yet Another Compiler-Compiler)

YACC is a parser generator tool. Given a grammar specification, YACC generates C code for an LALR(1) parser. BISON is the GNU version of YACC.

SectionContent
Declarations (%{ %})Token definitions (%token), type declarations, C includes
Grammar Rules (%%)production : body { semantic action }; where action is C code
C Functions (%%)Supporting C code; includes main() and yyerror()
💡 Key Variables:

$$ = value of left side (result)  |  $1, $2 = values of components  |  yyparse() = call parser  |  yyerror() = error function

UNIT 4

Syntax-Directed Translation

Attribute Grammars • S-Attributed • L-Attributed • Annotated Parse Trees • DAGs

4.1 Introduction

Syntax-Directed Translation (SDT) is a method of associating semantic actions (code) with grammar productions. Each production rule is augmented with a semantic rule (or action) that computes attribute values and can produce intermediate code.

📌 SDD

A context-free grammar together with attributes and rules. Each attribute is associated with a grammar symbol; rules specify how attribute values are computed.

📌 Synthesised Attribute

An attribute whose value is computed from attributes of its children in the parse tree. Computed bottom-up.

📌 Inherited Attribute

An attribute whose value is computed from attributes of its parent or siblings in the parse tree. Computed top-down.

📌 Annotated Parse Tree

A parse tree with attribute values written at each node, showing the results of semantic evaluation.

4.2 Attribute Grammars

ConceptDefinitionExample
Synthesised AttributeDefined from attributes of children; computed bottom-upE.val = E₁.val + T.val (for E → E + T)
Inherited AttributeDefined from parent/sibling attributes; computed top-downT.type = E.type (passing type info down)
Annotated Parse TreeParse tree with attribute values written at each nodeE.val=14, T.val=10, F.val=5 at each node
Dependency GraphShows dependencies between attribute instances; used to order evaluationNode B depends on Node A if B is computed from A

4.3 S-Attributed and L-Attributed Definitions

TypeDefinitionParser SuitabilityExample
S-Attributed SDDUses only synthesised attributes; all rules compute attributes from childrenBottom-up parsers (LR); actions at end of ruleE → E + T { E.val = E₁.val + T.val }
L-Attributed SDDSynthesised attributes + inherited attributes whose value depends only on left siblings or parent (left-to-right ordering)Top-down parsers (LL); both LL and LR with careD → T L { L.inh = T.type }
🧠 MEMORY TRICK:

'S = Synthesised-only (Simple, Bottom-up)' | 'L = Left to right (also inherited, Left siblings or parent only)'

4.4 Evaluation of SDDs

SchemeDescription
Dependency GraphA directed graph where nodes = attribute instances; edge from b to c means c depends on b
Topological SortAny ordering of dependency graph nodes consistent with edges; gives valid evaluation order
Parse-Tree OrderFor S-attributed: evaluate in post-order (children before parents); always works
Circular DependencyIf dependency graph has a cycle — SDD is not well-defined; no valid evaluation order

4.5 Syntax-Directed Translation Schemes (SDTs)

SDT PlacementWhen ExecutedBest For
Action at end of productionAfter all RHS symbols recognisedS-attributed definitions (most common)
Action in middle of productionWhen parser reaches that point in parsingL-attributed definitions; generates code during parsing
Action at start of productionBefore any RHS symbols recognisedInitialization; inherited attribute computation
💡 SDT for Desk Calculator:

E → E + T { print('+') } | T   T → T * F { print('*') } | F   F → num { print(num.val) }

4.6 Directed Acyclic Graphs (DAG) for Expressions

A DAG is a compact representation of an expression where common subexpressions are shared (represented by a single node). DAGs save space and reveal opportunities for optimisation.

FeatureParse Tree / ASTDAG
Common subexpressionsRepeated — separate nodesShared — single node with multiple parents
Space efficiencyMore space (full tree)Less space (sharing)
UseSyntax analysis, semantic checksOptimisation — common subexpression elimination
CreationOne node per operator/operand occurrenceReuse node if same operator and same operands

4.7 Translation of Declarations

VariableRole in Offset Calculation
offset (global)Tracks the next available memory offset; initialised to 0 for each scope
T.widthThe width (bytes) of type T: int=4, float=8, bool=1, pointer=8
id.offsetThe memory offset assigned to variable id when declared

5.1 Semantic Analysis

Semantic Analysis is the third phase of the compiler. It checks the program for semantic errors — errors that are syntactically valid but logically incorrect — and gathers type information needed for code generation.

Semantic CheckDescriptionExample Error
Type CheckingEnsures operations are applied to compatible typesint x = 3.14 without cast
Undeclared VariablesEnsures variables are declared before useUsing 'y' without int y;
Scope RulesEnsures variables are used within their valid scopeAccessing local variable outside function
Array Bound ChecksEnsures indices are valid (static check)int a[5]; a[10] = 1
Function Argument MatchingNumber and type of arguments match declarationadd(1, 2, 3) for add(int, int)
Return TypeReturned value matches declared return typeReturning float from int function
Duplicate DeclarationsVariables not declared more than once in same scopeint x; int x; in same block

5.2 Type Systems

TermDefinition
Type ExpressionA description of a type: basic type (int, float), constructor (array, pointer, record)
Type Equivalence (Name)Two types are equal only if they have the same name; Pascal-style
Type Equivalence (Structural)Two types are equal if their structures are identical; C-style
Widening ConversionAutomatic implicit conversion from smaller to larger type: int → float → double
Narrowing ConversionRequires explicit cast; possible data loss: double → int
CoercionImplicit type conversion inserted by compiler (e.g., int to float in mixed expression)
PolymorphismFunctions/operators work on multiple types: overloading, generics
💡 Type Widening Order:

bool → char → short → int → long → float → double — each conversion is widening (safe, no data loss)

5.3 Symbol Table — Detailed

ImplementationDescriptionTime Complexity
Linear ListSimple array or linked list of entriesO(n) lookup — too slow for large programs
Hash TableHash function maps identifier → bucket; most commonO(1) average lookup
Binary Search TreeSorted tree of entriesO(log n) lookup
Scope Stack (Chaining)Stack of hash tables; one per scope; look up through stackO(depth) worst case
💡 Scope Management:

On entering a new scope: push new symbol table. On exit: pop the table. Lookup searches from innermost to outermost scope.

5.4 Intermediate Code Generation

💡 Benefit:

With IR: N source languages + M target machines = N+M translators. Without IR: N×M translators needed!

IR FormDescriptionUse Case
Three-Address Code (TAC)Each instruction has at most one operator and three operands: x = y op zMost popular; easy optimisation

6.1 Introduction

Runtime environments deal with how the compiler organises memory during program execution. The compiler must generate code that manages data storage, function calls, and parameter passing correctly at runtime.

6.2 Storage Organisation

💡 Memory Layout (low to high address):

Code | Static Data | Heap (grows ↑) | ... | Stack (grows ↓)

6.3 Activation Records (Stack Frames)

An activation record (or stack frame) is the block of memory allocated on the call stack when a function is called. It stores all information needed to execute that function and return to the caller.

FieldContentNotes
Return ValueSpace for the function's return valueMay be in register instead
Actual ParametersCopies of or references to arguments passed by callerEvaluated before call
Control Link (Dynamic Link)Pointer to activation record of callerRestores stack on return

Saved Machine StatusRegisters and program counter saved before callRestored on return
Local VariablesVariables declared inside the functionAllocated in frame
TemporariesIntermediate values during expression evaluationCompiler-generated
💡 Stack Pointers:

SP (Stack Pointer) = top of stack | FP/BP (Frame Pointer/Base Pointer) = base of current activation record. Local var at FP + offset.

UNIT 7

Code Optimisation

Loop • DAG • Peephole • Data Flow • CSE • Constant Folding • Dead Code

7.1 Introduction

Code Optimisation is the phase that transforms the intermediate code to produce equivalent code that runs faster, uses less memory, or consumes less energy — without changing the program's output. It is the most complex phase of a compiler.

📌 Key Principle:

Optimisation MUST preserve program semantics — the output of the optimised program must be identical to the original for all valid inputs.

Optimisation TypeWhere AppliedDescription
Local OptimisationWithin a Basic BlockTransformations within a straight-line code sequence; no branches
Global OptimisationWithin a ProcedureTransformations spanning multiple basic blocks within a function
InterproceduralAcross ProceduresOptimisations considering multiple functions together; inline expansion
Loop OptimisationWithin LoopsCritical — most execution time spent in loops; code motion, induction variables
Machine-SpecificTarget MachinePeephole, register allocation, instruction scheduling

7.2 Basic Blocks & Flow Graphs

TermDefinition
Basic BlockMaximal straight-line code sequence; single entry, single exit; no branches in the middle
LeaderFirst statement of a basic block; identified by: (1) first statement, (2) target of a branch, (3) statement after a branch
Flow GraphA directed graph where nodes = basic blocks; edges = possible control flow between blocks
PredecessorBlock B is a predecessor of C if there is a flow graph edge B→C
SuccessorBlock C is a successor of B if there is a flow graph edge B→C

7.3 Local Optimisations (within Basic Blocks)

OptimisationDescriptionExample (Before → After)
Constant FoldingEvaluate constant expressions at compile timex = 2 * 3 + 4 → x = 10
Constant PropagationReplace variable with its known constant valuex=5; y=x+3 → y=8
Common Subexpression Elimination (CSE)Reuse already-computed value instead of recomputingt1=a+b; t2=a+b → t1=a+b; t2=t1
Dead Code EliminationRemove code whose results are never usedx=5; x=6; use(x) → x=6; use(x)
Algebraic SimplificationApply algebraic identities to simplifyx = y * 1 → x = y
Strength Reduction (local)Replace expensive operations with cheaper equivalentsx = y * 2 → x = y + y
Copy PropagationReplace a copy x=y with uses of y directlyx=y; z=x+1 → z=y+1

7.4 Loop Optimisations

Loops are the most critical targets for optimisation because code inside loops is executed repeatedly. Even small improvements yield large runtime gains.

Loop OptimisationDescriptionExample
Loop-Invariant Code MotionMove computation whose result doesn't change across iterations outside the loopfor(i=0;i<n;i++) x=y*z+i → precompute t=y*z; loop uses t
Induction Variable EliminationSimplify computations on variables that change by constant amounts each iterationReplace j=4*i (expensive multiply) with j+=4 (cheap add)
Strength Reduction (Loop)Replace multiply by induction var with cheaper additiont1=4*i in loop → t1 starts at 0, incremented by 4 each iteration
Loop UnrollingReplicate loop body multiple times; reduces loop overheadfor(i=0;i<4;i++) A[i]=0 → A[0]=0; A[1]=0; A[2]=0; A[3]=0
Loop FusionCombine two adjacent loops with same bounds into oneReduces loop overhead; improves cache locality
Loop InterchangeSwap nested loop order for better memory access patternImproves cache efficiency for 2D array traversal
🧠 MEMORY TRICK:

'LISSU' — Loop-Invariant, Induction vars, Strength reduction, Split/fusion, Unrolling — 5 loop optimisations!

7.6 Peephole Optimisation

Peephole optimisation examines a small sliding window (the 'peephole') of the generated target code and replaces sequences with shorter or faster equivalents.

CategoryPatternOptimised
Redundant Load/StoreSTORE X; LOAD XRemove LOAD if X not changed
Unreachable CodeJUMP L1; [code between]; L1:Remove unreachable code after jump
Flow of ControlJUMP L1; ... L1: JUMP L2Replace with JUMP L2 directly

Strength Reductionx = x * 2Replace with x = x + x or x = x << 1

UNIT 8

Code Generation

Target Code • Register Allocation • Instruction Selection • Basic Block Code Gen

8.1 Introduction

Code Generation is the final phase of the compiler. It translates the (optimised) intermediate representation into target machine code. The target is usually assembly language or machine code for a specific architecture.

📌 Code Generator:

Takes an IR as input and produces target code. Must correctly map IR operations to machine instructions, allocate registers, and manage memory access.

Code Generator IssueDescription
Instruction SelectionChoose the best machine instructions to implement each IR operation; one IR op may map to several choices
Register AllocationDecide which values to keep in registers; registers are scarce, fast resources
Register AssignmentAssign specific hardware registers to values chosen for register storage
Evaluation OrderThe order in which computations are performed affects register usage and efficiency

8.2 Target Machine Model

For our discussion, we use a simple register machine model with n general-purpose registers R0, R1, ..., Rn-1.

Instruction ClassFormatDescription
LoadLD Ri, MLoad from memory M into register Ri
StoreST M, RiStore from register Ri into memory location M
ComputationOP Ri, Rj, RkRi = Rj OP Rk (e.g., ADD R1, R2, R3)
Unconditional BranchBR LJump to label L
Conditional BranchBCond Ri, LJump to L if Ri satisfies condition
Indexed AddressLD Ri, a(Rj)Ri = Memory[a + Rj] (array access)

8.3 Instruction Selection

IR OperationNaive Target CodeBetter Target Code
a = b + cLD R0,b / LD R1,c / ADD R0,R0,R1 / ST a,R0If b in register: ADD Rb, Rb, c (3 instr → 1)
a = b * 8LD R0,b / MUL R0,R0,#8 / ST a,R0LD R0,b / SHL R0,R0,#3 / ST a,R0 (shift is faster)
a = a + 1LD R0,a / ADD R0,R0,#1 / ST a,R0INC a (single instruction on some architectures)
a = -bLD R0,b / NEG R0 / ST a,R0Direct NEG instruction if available

8.4 Register Allocation & Assignment

Registers are the fastest storage but are limited in number. Good register allocation is critical to performance — memory accesses are 100x slower than register operations.

ConceptDescription
Register AllocationDecide which program variables/temporaries should be kept in registers at each point
Register AssignmentDecide which specific register (R0, R1, ...) holds each allocated value
SpillingWhen all registers are full, a value is written to memory (spilled) to free a register
Live Variable AnalysisUsed to determine which values still need to be in registers; dead values can be evicted
Graph ColouringModel allocation as graph colouring; variables are nodes, edges for conflicts; colours = registers

8.4.1 Register Allocation using Graph Colouring

Chaitin's algorithm uses interference graphs for register allocation:

StepAction
1. Build Interference GraphNodes = variables/temporaries; add edge between x and y if both live at same point
2. Colouring AttemptTry to colour graph with k colours (k = number of registers); no adjacent nodes share colour
3. If k-colourableRegister assignment is: variable's colour = assigned register
4. If not k-colourableSelect a variable to spill to memory; remove node; retry colouring
5. Assign RegistersMap each colour to an actual hardware register

8.5 Code Generation for Basic Blocks — getReg Algorithm

The getReg algorithm (from Aho et al.) decides which register to use for each operand in a TAC instruction within a basic block.

DescriptorRecords
Register DescriptorWhich values are currently in each register
Address DescriptorWhere each variable's current value can be found (register, memory, or both)
StepgetReg Logic
1. Check address descriptorIf the required value is already in a register, use that register
2. Find free registerIf a register is free (not holding any live value), allocate it
3. Spill a registerIf no free register, choose a register whose value is (a) not live after this block, or (b) also in memory; spill if needed
4. Update descriptorsAfter each instruction, update both register descriptors and address descriptors
💡 Solved Example — Code Gen for d = (a - b) + (a - c):

TAC: t1=a-b; t2=a-c; d=t1+t2

Code: LD R0,a / LD R1,b / SUB R0,R0,R1 [R0=t1] / LD R1,c / SUB R1,R0,R1 [R1=t2, R0=a-b reused!] / ADD R0,R0,R1 [R0=d] / ST d,R0

8.6 Special Considerations in Code Generation

TopicDescription
Instruction SchedulingReorder instructions to minimise pipeline stalls; must respect data dependencies
Calling ConventionsRules for how functions pass arguments/return values (caller-save vs callee-save registers)
Inline ExpansionReplace function call with function body; eliminates call overhead but increases code size
Branch Delay SlotsRISC processors: instruction after branch always executes; schedule useful work there
